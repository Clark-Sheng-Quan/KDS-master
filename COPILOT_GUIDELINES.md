# Copilot 工作规范

## 核心原则

### 1. 禁止创建文档文件
- **规则**：不要创建任何 `.md` 文档、总结或说明文件
- **例外**：除非明确要求创建项目文档
- **原因**：保持代码库整洁，信息通过对话交流
- **违反后果**：删除所有非必要的 md 文件

### 2. 重大改动需预先通知
- **规则**：在进行长时间的或重大改动之前，必须先向用户询问并获得确认
- **触发条件**：
  - 涉及 3 个以上文件的改动
  - 预计耗时 10 分钟以上
  - 改变现有功能的实现逻辑
  - 重构代码结构或架构
  - **所有 Java 文件改动**（即使只涉及 1-2 个文件）
  
- **通知内容必须包括**：
  - 计划改动的具体内容
  - 涉及的文件列表
  - 预计耗时
  - 可能的影响范围
  - 示例："我发现需要重构订单状态管理，涉及 OrderContext.tsx、OrderService.ts、OrderCard.tsx 三个文件，预计 20 分钟，会改变订单状态更新的实现方式。确认要开始吗？"

- **执行流程**：
  1. 停止工作，通知用户
  2. 等待用户确认（"好"、"可以"、"开始"、"同意" 等）
  3. 收到确认后才开始执行改动
  4. 不要未经许可就开始大规模修改

---

## 工作流程

### 发现问题/需求
1. 分析问题
2. 制定解决方案
3. 评估改动规模

### 规模评估
- **小改动** ✅ 直接执行
  - 单文件修改
  - 简单 bug 修复
  - 添加小功能
  
- **中等改动** ⚠️ 通知用户后执行
  - 2-3 个文件涉及
  - 功能逻辑改变
  - 预计 10-20 分钟
  
- **重大改动** 🛑 必须确认后执行
  - 3 个以上文件
  - 架构重构
  - 预计 20+ 分钟
  - 影响多个功能模块

## 常见场景

| 场景 | 处理方式 |
|-----|--------|
| 修复单个文件的 bug | 直接修复 ✅ |
| 添加简单功能到单个组件 | 直接添加 ✅ |
| 修改 Java 文件（任何改动） | 先通知 🛑 |
| 修改 3+ 个 React/TypeScript 文件 | 先通知 🛑 |
| 重构订单流程 | 先通知 🛑 |
| 调整 UI 样式（单文件） | 直接修改 ✅ |
| 修改 API 调用方式 | 先通知 🛑 |
| 测试代码修改 | 直接修改 ✅ |
| 重构状态管理系统 | 先通知 🛑 |
